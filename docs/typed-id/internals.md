# Internals

This page documents how `[TypedId]` is implemented. You don't need any of this to use the library, but it's useful if you're debugging a generator issue, checking whether a code path allocates, or contributing.

## Generator pipeline

`TypedIdGenerator` is a Roslyn `IIncrementalGenerator` living in the existing `ZeroAlloc.ValueObjects.Generator` project. It runs in parallel with the existing `ValueObjectGenerator` and shares Roslyn utility code, but neither parser nor writer is shared.

```mermaid
flowchart LR
    A([Source files]) --> B[IIncrementalGenerator<br/>syntax filter:<br/>record struct + [TypedId]]
    B --> C[Symbol-level<br/>semantic check]
    C --> D{Valid?}
    D -->|No| E[Emit ZATI001/002/003/005]
    D -->|Yes| F[Resolve Strategy + Backing<br/>per-struct → assembly → default]
    F --> G[SourceWriter]
    G --> H([StructName.TypedId.g.cs])
```

Key properties:

- **Incremental** — only re-runs for structs whose syntax or attribute values changed. IDE edits stay fast.
- **No reflection at generation time** — all metadata comes from the Roslyn `ISymbol` API.
- **One file per struct** — easier to debug; the generator cache tracks each file independently.

## Resolution order

For every `[TypedId]` struct the generator computes `(Strategy, Backing)` in this order:

1. `[TypedId(Strategy = ..., Backing = ...)]` on the struct itself.
2. `[assembly: TypedIdDefault(Strategy = ..., Backing = ...)]`.
3. Built-in: `Strategy = Ulid`, `Backing = Default`.

`BackingType.Default` is then resolved to a concrete backing:

- `Ulid` / `Uuid7` → `Guid`
- `Snowflake` / `Sequential` → `Int64`

Incompatible combinations (`Snowflake + Guid`, `Ulid + Int64`) raise [ZATI001](diagnostics.md) and suppress code generation for that struct.

## What gets emitted

The generated file for an `OrderId` (Ulid + Guid) contains, in order:

1. File header with `<auto-generated>` marker (so analyzers ignore it).
2. `using` directives for `System`, `System.Text.Json`, `System.Text.Json.Serialization`, and the runtime-helper namespace.
3. The partial declaration carrying `[JsonConverter(typeof(JsonConv))]` and interface list.
4. `public Guid Value { get; }` and the constructor.
5. `New()`, `Parse`/`TryParse` (both `string` and `ReadOnlySpan<char>` overloads), `ToString()`, `CompareTo`.
6. Private sealed nested `JsonConv : JsonConverter<OrderId>`.

For Int64-backed strategies the shape is identical with `long Value` and decimal-string parsing. Neither variant emits `Equals` / `GetHashCode` / `==` / `!=` — `record struct` gives those for free over the single `Value` field.

The generator source is at [`src/ZeroAlloc.ValueObjects.Generator/TypedIdGenerator.cs`](https://github.com/ZeroAlloc-Net/ZeroAlloc.ValueObjects) in the repository.

## Zero-allocation techniques

The runtime helpers (`UlidCore`, `Uuid7Core`, `SnowflakeCore`, `SequentialCore`) are in the core package and use only BCL primitives. The hot paths are:

### `stackalloc` for transient buffers

ULID encoding needs a 16-byte random buffer (the Guid bytes) and a 26-char output buffer. Both are `stackalloc`:

```csharp
// Simplified UlidCore.ToBase32 sketch
public static string ToBase32(Guid value)
{
    Span<byte> bytes = stackalloc byte[16];
    value.TryWriteBytes(bytes);
    Span<char> chars = stackalloc char[26];
    EncodeCrockford(bytes, chars);
    return new string(chars);     // the only allocation is the final string
}
```

`TryParse` goes the other direction with no allocations at all — the caller's `ReadOnlySpan<char>` stays on the stack, decoded into the outgoing `Guid`.

### `[ThreadStatic]` scratch for RNG

`RandomNumberGenerator.Fill` itself is already alloc-free. ULID / UUIDv7 generation uses a thread-static buffer only when assembling the full 128-bit value before packing into a `Guid`.

### `Interlocked.CompareExchange` for lock-free state

`SnowflakeCore` packs the last observed `(timestamp_ms, sequence)` into a single `long`:

```
 bits 63..22: timestamp_ms (last observed)
 bits 21..10: reserved (0)
 bits  9..0:  sequence (reused as per-ms counter)
```

On every `Next()` call, the core CAS-loops on this packed state:

```csharp
private static long _state;   // packed (timestamp_ms, sequence)

public static long Next()
{
    while (true)
    {
        var nowMs  = UnixMs();
        var snap   = Volatile.Read(ref _state);
        var next   = ComputeNext(snap, nowMs);       // pure function
        if (Interlocked.CompareExchange(ref _state, next, snap) == snap)
            return Pack(next, workerId: TypedIdRuntime.WorkerId);
    }
}
```

No locks, no `lock` statement, no thread-affine state. Concurrent `New()` calls from many threads contend on the same 8-byte word and either advance the millisecond or increment the sequence.

### No boxing on JSON path

The generated `JsonConv` calls `writer.WriteStringValue(value.ToString())` and `OrderId.Parse(reader.GetString()!)`. The only unavoidable allocation is the string itself — the BCL's `Utf8JsonWriter` has no overload for arbitrary spans without materializing a string.

`Utf8JsonReader.GetString()` allocates; reading UTF-8 directly via `ValueSpan` avoids that. A future version may use the UTF-8 path for hot deserialize scenarios; v1 uses the simpler string path.

### Sequential

`SequentialCore` is a single `Interlocked.Increment(ref _counter)`. Zero allocations, no worker ID, no clock.

## JsonConverter pattern

Each `[TypedId]` struct has a nested `private sealed class JsonConv : JsonConverter<T>`. The struct itself carries `[JsonConverter(typeof(JsonConv))]`. `System.Text.Json` discovery:

1. `JsonSerializer` sees `[JsonConverter]` on the type.
2. Instantiates `JsonConv` (parameterless, so no DI, no factory).
3. Calls `Read` / `Write` on the instance.

Because the converter type is known at compile time, `PublishAot` and `JsonSerializerContext` both see it as an ordinary converter — no reflection emit, no dynamic type resolution.

The nesting keeps the converter invisible in the type's public API and prevents accidental construction from user code.

## `IParsable<T>` integration with ASP.NET

ASP.NET Core's minimal-API binder iterates a fixed list of parameter kinds. For complex types it checks, in order:

1. `TryParse(string, IFormatProvider?, out T)` — the `IParsable<T>` shape.
2. `BindAsync(HttpContext, ParameterInfo)` — async binder.
3. Constructor with `(string)` parameter.
4. Fall back to `[FromBody]` JSON.

The generated `TryParse` sits at #1. The binder calls it with the raw route / query token; a `false` return translates to 400. No registration, no middleware, no options.

For span-based routes (the framework uses spans where the URI is already available as one), `ISpanParsable<T>.TryParse` is preferred — zero-alloc binding end-to-end.

## EF Core convention via reflection

`AddTypedIdConventions` uses reflection **once** at `ModelConfigurationBuilder` time, not per operation. The hot path is:

1. At convention-build (model-build): scan the target assembly for `[TypedId]` structs.
2. For each, instantiate `TypedIdValueConverter<TId, TBacking>` and register via `builder.Properties<TId>().HaveConversion<...>()`.
3. EF Core compiles the converter into its internal query/materialization pipeline and caches the result in the model.

Per-query and per-row code paths do not reflect — they invoke the cached converter directly. The converter itself is a trivial `TId -> TBacking` lambda over `Value`, which the runtime JITs into a field read on the struct and a field assignment on the other side. No boxing, no allocations beyond what EF Core already does.

## Testing invariants

The test suite enforces:

- Verify-based snapshot tests: one per `(strategy × backing × scenario)` combination.
- Diagnostic tests: one per `ZATI###` asserting position and message template.
- Allocation tests (BenchmarkDotNet + `[MemoryDiagnoser]`): CI fails the build if `New()` or `TryParse(ReadOnlySpan<char>)` allocates any bytes on any strategy.
- Concurrency: 16 threads × 10k `Sequential.New()` calls must yield 160 000 strictly distinct values.
- EF Core: SQLite in-memory round-trip per backing, plus a convention-vs-override test.

If you're contributing and your change touches runtime code, re-run the benchmark project locally to confirm the zero-alloc guarantee holds.
